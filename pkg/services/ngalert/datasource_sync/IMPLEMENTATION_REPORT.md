# Implementation Report: Automatically Sync Mimir Alertmanager Datasource Configuration

**Ticket:** gra-yllc
**Date:** 2026-03-09
**Parent:** war-hcc1 (single Alertmanager, stage 3)

---

## Overview

This feature adds a background worker that periodically fetches the Mimir Alertmanager configuration from a configured datasource and automatically mirrors it as an `ExtraConfiguration` in Grafana. It removes the need for a manual two-step import process required by SLO, Asserts, and similar integrations.

## Decisions

### 1. Polling interval: open question

The current implementation reuses `AdminConfigPollInterval` (default 1 min) from `UnifiedAlertingSettings`. The rationale was that both the `AlertsRouter` and the datasource sync worker reconcile external alertmanager state on a regular cadence. However this coupling is questionable: `AdminConfigPollInterval` controls how often Grafana checks *which orgs are routing alerts to external alertmanagers* — a delivery concern. Datasource sync is a *configuration mirroring* concern with a different failure mode and arguably a different desired frequency.

A cleaner alternative is a dedicated `DatasourceSyncPollInterval` field in `UnifiedAlertingSettings`, defaulting to 1 minute. This decouples the two concerns and gives operators independent control. The current implementation used `AdminConfigPollInterval` as a first-pass approximation; this should be revisited before the feature is generally available.

### 2. Why a DB table, and what "one sync config per org" means

The `ngalert_datasource_sync` table stores one *sync configuration record* per Grafana organisation. Each record holds:
- `datasource_uid` — which Mimir Alertmanager datasource to poll
- `enabled` — whether syncing is currently active for this org
- `last_sync_at` / `last_error` — operational status surfaced to operators via the API

**Why store this in the DB at all?** The alternative is static Grafana config (`.ini`), but that would require a restart to change and cannot be scoped per org in a multi-tenant setup. A DB row allows an admin to enable/disable sync at runtime via the REST API and survives restarts.

**Why one row per org?** This is a direct consequence of `SaveAndApplyExtraConfiguration` already enforcing a single `ExtraConfiguration` per org — there is nowhere to put a second synced config even if one were configured. The unique index on `org_id` makes this constraint explicit at the DB level rather than relying on application-layer enforcement. If multi-datasource sync is introduced in the future, both the table schema and `SaveAndApplyExtraConfiguration` would need to change together.

### 3. REST API routes

Two routes were added, both admin-only and behind the feature flag:

- `GET /api/v1/ngalert/datasource_sync` — returns the current sync configuration and operational status (datasource UID, enabled flag, last sync time, last error) for the calling org. Returns `{"enabled": false}` when no configuration exists yet.
- `POST /api/v1/ngalert/datasource_sync` — enables or disables sync and sets the target datasource UID for the calling org.

The `POST` body accepts any datasource UID without type-checking. The import UI constrains selection to `type === 'alertmanager'` datasources via `getAlertManagerDataSources()` in `public/app/features/alerting/unified/utils/datasource.ts`. The API should mirror this by validating that the supplied UID refers to an Alertmanager-type datasource before persisting it. This is an **open gap**.

**Why a REST API and not static config?** The primary value is runtime configurability without a Grafana restart, and per-org scope in multi-tenant deployments. The `GET` response also doubles as a status endpoint, exposing the last sync time and any error message — important for operators debugging why a sync is failing.

**On Swagger codegen:** The existing ngalert routes are defined in Swagger-generated files. Adding new routes there requires regenerating the spec and running codegen. This was not done here — routes are registered via a hand-written `registerDatasourceSyncEndpoints` method following the same pattern as `RegisterHistoryApiEndpoints`. Integrating with codegen is a follow-up.

### 4. Basis for the conversion logic: `RouteConvertPrometheusPostAlertmanagerConfig`

The direct model for how the worker consumes `SaveAndApplyExtraConfiguration` is `RouteConvertPrometheusPostAlertmanagerConfig` in `pkg/services/ngalert/api/api_convert_prometheus.go` (line 638). That handler:

1. Receives a Mimir-format config body (`AlertmanagerUserConfig`) from an inbound HTTP request
2. Wraps it into an `ExtraConfiguration` struct with an identifier, merge matchers, template files, and the raw alertmanager YAML
3. Validates and calls `SaveAndApplyExtraConfiguration(ctx, orgID, ec, replace, dryRun)`

The worker does the same thing driven by a timer and an outbound HTTP `GET`, but there are two meaningful differences from how the import UI calls this handler.

**Identifier:** The handler reads the identifier from the `X-Grafana-Alerting-Config-Identifier` request header, defaulting to the string `"imported"` if absent. In the import UI (`Step1AlertmanagerResources.tsx`) the user supplies a `policyTreeName` (a human-readable label, validated against RFC 1123 subdomain rules, max 40 chars). The worker uses the datasource UID as the identifier. This is stable and unique per org, but diverges from the UI convention and may create inconsistency if both paths are used on the same org.

**Merge matchers vs. ManagedRoutes:** The import UI always sends `X-Grafana-Alerting-Merge-Matchers: __grafana_managed_route__=<identifier>`. This populates `ExtraConfiguration.MergeMatchers`, which is passed as `SubtreeMatchers` into `definition.Merge` in the `grafana/alerting` library. Reading `definition/merge.go` directly: **if `SubtreeMatchers` is empty, `definition.Merge` skips routing tree merging entirely** — only receivers and time intervals are deduplicated.

However, there are two separate scoping mechanisms in play:

- **Old approach (`MergeMatchers` / `SubtreeMatchers`):** The imported routing tree is embedded directly into the merged routing tree with the matcher applied by `definition.MergeRoutes`. This path is active regardless of feature flags.
- **New approach (`ManagedRoutes` + `FlagAlertingMultiplePolicies`):** `GetMergedAlertmanagerConfig` always returns `ExtraRoute` (the imported routing tree root). When `FlagAlertingMultiplePolicies` is enabled, `alertmanager_config.go` (line 125–139) takes that `ExtraRoute` and adds it to `ManagedRoutes`, then calls `WithManagedRoutes` which injects `__grafana_managed_route__=<identifier>` automatically — no `MergeMatchers` needed.

The worker leaving `MergeMatchers` unset is correct. It calls `SaveAndApplyExtraConfiguration` → `modifyAndApplyExtraConfiguration` → `am.SaveAndApplyConfig`, and `BuildNotificationsConfiguration` — which contains the `WithManagedRoutes` / `FlagAlertingMultiplePolicies` logic — is part of that apply path. The worker reuses the same mechanism as every other caller; the routing tree scoping behaviour with or without `FlagAlertingMultiplePolicies` is identical to what any caller of `SaveAndApplyExtraConfiguration` with empty `MergeMatchers` would observe.

### 5. Package placement: `pkg/services/ngalert/datasource_sync/`

The worker was placed in its own package rather than inside `sender/` or `notifier/`. This mirrors how `sender/router.go` (`AlertsRouter`) is a discrete concern with its own package boundary. Keeping datasource sync isolated avoids coupling it to alert routing internals and makes the package independently testable with a narrow set of dependencies.

### 6. `ExtraConfigApplier` interface defined locally in the worker package

Rather than importing `*notifier.MultiOrgAlertmanager` directly, the worker depends on a locally-defined `ExtraConfigApplier` interface with only the `SaveAndApplyExtraConfiguration` method. This makes the worker independently testable via a simple mock and avoids importing the heavy `notifier` package into the worker package's test graph.

### 7. Worker is always constructed; feature flag gates execution

The `Worker` is unconditionally constructed in `ngalert.go` and its goroutine is always started in `Run()`. The feature flag check is at the top of `Worker.Run()`, causing it to return immediately when disabled. This pattern matches how other workers in ngalert behave and avoids nil-guarding the worker reference throughout the codebase.

### 8. Auth reuses `datasourceService.CustomHeaders` + basic auth from `SecureJsonData`

The worker replicates the auth-building steps from `sender/router.go`'s `datasourceToExternalAMcfg`:
- Custom headers via `datasourceService.CustomHeaders` (covers `X-Scope-OrgID`, API tokens, etc.)
- Basic auth via `secretService.GetDecryptedValue` on `SecureJsonData`

TLS client cert support was not included. It can be added in a follow-up; the current scope targets the most common Mimir authentication patterns.

### 9. Config fetch endpoint: `GET /alertmanager/api/v1/alerts`

Mimir exposes its alertmanager configuration at `/alertmanager/api/v1/alerts` — the same endpoint `mimirtool alertmanager get` targets and that Grafana's own `/convert/api/v1/alerts` route mirrors. The worker's `buildConfigURL` replicates the path-adjustment logic from `sender/router.go`'s `buildExternalURL` (appending `/alertmanager` for `mimir`/`cortex` implementations) and appends `/api/v1/alerts`.

### 10. Per-sync errors are recorded and do not abort the full sync pass

In `SyncAll`, errors from individual orgs are logged and recorded in the DB via `UpdateDatasourceSyncStatus`, but do not cause `SyncAll` to return an error. This mirrors `AlertsRouter.SyncAndApplyConfigFromDatabase`, which processes all orgs and logs individual failures. The `last_error` column surfaces the most recent error via `GET /api/v1/ngalert/datasource_sync`.

### 11. `GET` response when no configuration exists

Returns `200 OK` with `{"enabled": false}` rather than `404 Not Found`. "No record" and "sync disabled" are semantically equivalent from the caller's perspective; a 404 would force callers to handle an extra status code for what is a normal state.

## Not Implemented (Out of Scope / Deferred)

| Item | Reason |
|------|--------|
| Dedicated `DatasourceSyncPollInterval` config field | Currently reuses `AdminConfigPollInterval`; decoupling is recommended before GA |
| Behaviour with `FlagAlertingMultiplePolicies` off | Routing tree is not embedded (same as any other `SaveAndApplyExtraConfiguration` caller with empty `MergeMatchers`); not specific to the worker |
| **Datasource type validation in `POST` handler** | **Open gap** — the API accepts any UID; should reject non-`alertmanager`-type datasources to match UI behaviour |
| **Identifier alignment with import UI convention** | **Open gap** — worker uses datasource UID; UI uses a user-supplied `policyTreeName`; inconsistency if both paths are used on the same org |
| Import UI disabling the Import button when sync is active | Frontend work; separate ticket expected |
| Integration test for end-to-end sync flow | Requires a live Mimir instance or integration test harness; marked as acceptance criterion but deferred |
| TLS client cert auth in the worker | Uncommon in typical Mimir setups; addable in a follow-up |
| Swagger codegen for new routes | Routes registered manually for now; codegen integration is a follow-up |
| Multi-datasource sync per org | `SaveAndApplyExtraConfiguration` enforces a single `ExtraConfiguration` per org; requires changes at both the API and schema level |
